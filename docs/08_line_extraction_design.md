# 08. 実線抽出パイプライン設計（OpenCV.js）

手のひら画像から **実際の手相線を抽出・測定** し、診断（`palm_rules.json`）へ
本物の特徴量を渡すための設計。これにより「占い結果の信ぴょう性」を担保する。

> 現状：線オーバーレイは MediaPipe ランドマークに合わせた**汎用テンプレートの追従配置**で、
> 特徴量は**モック生成（ランダム）**。本設計はこれを実測に置き換える（②）。
> 既存：`lib/handDetect.ts`(検出/正規化) / `lib/palmLines.ts`(線の点列) / `ResultStep`(描画/手調整③)。

---

## 全体フロー

```
正規化画像(EXIF焼込済) + 21ランドマーク
  │
  ▼
[1] パームROIの矩形化(rectify)      … 指の谷+手首で座標正規化 → 正準画像 P(固定サイズ)
  │
  ▼
[2] 前処理                          … グレースケール / 照明補正 / CLAHE / 軽い平滑化
  │
  ▼
[3] 線強調(谷=暗い溝の検出)         … 方向別 black-hat / Gabor バンク → 線強調画像 L
  │
  ▼
[4] 二値化・整形                    … adaptiveThreshold → 形態学 → 微小成分除去
  │
  ▼
[5] 線ごとのROIで候補抽出・曲線当て  … 帯ROI×期待方向で成分選択 → 順序化 → 多項式/スプライン
  │
  ▼
[6] 逆変換で元画像座標へ            … 正準画像→元画像の逆warpで制御点列に戻す
  │
  ▼
[7] 特徴量の実測                    … length/curve/depth/slope/start_height/presence を測る
  │
  ▼
[8] 診断へ                          … diagnoseHand に実測特徴を渡す（モック置換）
                                       低信頼は standard 扱い（断定しない）。手調整③は補正として併存
```

`[1]` は前段調査の **指の谷ベース座標系**（docs/07 追補）を使う。
`[3]–[5]` が画像処理の難所。`[7]–[8]` は既存ロジックにそのまま接続できる。

---

## [1] パームROIの矩形化（rectify）

生体認証(palmprint)の定番に倣い、**指の谷2点＋手首**でパーム座標系を作り、
固定サイズの正準画像へ warp する（スケール・回転・傾きを正規化）。

- 基準点（MediaPipe）：
  - `A = (lm5 + lm9)/2`  … 人差し指‑中指の谷（radial-top）
  - `B = (lm13 + lm17)/2` … 薬指‑小指の谷（ulnar-top）
  - `W = lm0` … 手首
- パーム座標 (u,v)：
  - u 軸 = `A→B`（横）、原点 u=0 を A、u=1 を B
  - v 軸 = `topMid(=(A+B)/2) → W`（縦＝手首方向）、v=0 が谷ライン、v=1 が手首
  - 点の写像：`P(u,v) = A + u·(B−A) + v·(W − topMid)`
- 正準画像 `S×S`（例 384）へのアフィン：u∈[−0.7, 1.5], v∈[0, 1.5] を画像全体へ写す
  3点対応で `cv.getAffineTransform` → `cv.warpAffine`。逆行列 `Minv` を保持（[6]で使用）。
  - ※ 生命線は親指側(負u)・手首(大v)へ伸びるためROIを広めに取る。
- 出力：正準パーム画像 `P`（グレースケール）、逆変換 `Minv`。

---

## [2] 前処理

- `cv.cvtColor(P, gray, COLOR_RGBA2GRAY)`
- 照明補正：大σの Gaussian で背景推定 → `gray − background`（陰影除去）
- `cv.createCLAHE(clipLimit=2.0, tile=8×8)` でしわのコントラスト強調
- 軽い平滑化（`cv.bilateralFilter` 弱め）で肌テクスチャのノイズ低減（線は保持）

---

## [3] 線強調（手相線＝暗い谷の検出）

手相線は周囲の肌より**暗く細い溝**。これを方向選択的に強調する。

- 第1候補：**方向別 black-hat**
  `cv.morphologyEx(gray, MORPH_BLACKHAT, lineKernel(θ))` を θ∈{0,30,…,150°} で実行し
  画素ごと最大応答。線状の構造要素（細長い長方形）で暗い線を抽出。
- 第2候補：**Gabor バンク**
  `cv.getGaborKernel(ksize, σ, θ, λ, γ)` を複数θで畳み込み、応答最大を採用。
  線幅 λ は正準画像の想定線幅に合わせる。
- 細かいしわはこの後 [4] の長さ・面積フィルタで除去（主要線だけ残す）。
- ROIごとに**期待方向**が分かるので（[5]）、θを絞ってクロストークを減らせる。

> OpenCV.js 可用API：`getGaborKernel` / `morphologyEx` / `filter2D` は利用可。
> `ximgproc.thinning`(細線化) は contrib のため標準ビルドに無い → [5] は多項式当てで回避。

---

## [4] 二値化・整形

- `cv.adaptiveThreshold(L, ADAPTIVE_THRESH_GAUSSIAN, THRESH_BINARY, block, C)`
- `cv.morphologyEx(OPEN→CLOSE)` で点ノイズ除去・小ギャップ連結
- `cv.connectedComponentsWithStats` で成分化 → **面積/長さが閾値未満を除去**

---

## [5] 線ごとのROIで候補抽出・曲線当て

正準フレームでの **帯ROI × 期待方向**（docs/07追補の位置情報に基づく）：

| 線 | ROI（v=谷ライン基準の縦割合 / u範囲） | 期待方向 |
|---|---|---|
| 感情線 | v≈0.12–0.25 の横帯 / u≈−0.1〜1.3 | 水平 |
| 知能線 | v≈0.30–0.50 の横帯 / u≈−0.3〜1.1 | 水平（やや右下がり） |
| 生命線 | 親指側の弧帯 / u≈−0.6〜0.1, v≈0.2–1.1 | 縦〜斜め（弧） |
| 運命線 | 中央の縦帯 / u≈0.3〜0.6, v≈0.2–1.1 | 垂直 |

- 各ROI内で、向きが合致する成分のうち**最も長く濃いもの**を主要線として選択。
- 成分点列を**主軸(PCA)順**に並べ、低次多項式（生命線は媒介変数曲線）でフィット
  → `Pt[]`（制御点列、正準座標）。細線化は使わず多項式当てで安定化。
- 信頼度：成分長/ROI被覆率/平均コントラストから 0–1 を算出。

---

## [6] 元画像座標へ逆変換

- 制御点列に `Minv` を適用 → 元画像（オーバーレイ viewBox と同座標系）へ。
- これが `palmLines` のテンプレ点列を**置き換える初期線**になる。
  手調整③は引き続き「補正レイヤ」として上に乗る。

---

## [7] 特徴量の実測（信ぴょう性の要）

フィットした各線から、`palm_rules.json` の enum 特徴を**測る**：

| 特徴 | 測り方（正準フレームで正規化済み） | enum |
|---|---|---|
| length | 線長 / パーム基準長 | long/standard/short |
| curve(生命線) | 弦からの最大乖離 / 弦長 | deep/standard/shallow |
| depth | 線画素の暗さ（周囲肌との輝度差・線幅） | dark/standard/faint |
| slope(知能線) | 始点→終点の角度 | downward/straight/upward |
| start_height(感情線) | 起点の縦位置 | high/standard/low |
| presence(運命線/extra3) | ROIで信頼度十分な線が在るか | present/absent |

- 閾値は実測キャリブレーション（複数サンプル）で決める。
- **低信頼は `standard`／`absent` 扱い**（standardはルールを持たない＝断定しない。CLAUDE.md準拠）。

---

## [8] 診断への接続

- `genMockFeatures()` を **実測特徴に置換**（検出成功時）。失敗・低信頼はモック/standardへフォールバック。
- 既存 `diagnoseHand` / `combos` / `advice` はそのまま機能（特徴量の出所が変わるだけ）。

## モジュール構成・実装方針

- 新規 `lib/lineExtraction.ts`（OpenCV.js 重処理）。返り値：
  ```ts
  type LineExtractResult = {
    points: Partial<Record<LineKey, Pt[]>>;     // 元画像座標
    features: Partial<Record<LineKey, Features>>;
    confidence: Partial<Record<LineKey, number>>;
  };
  ```
- OpenCV.js は大きい(~8MB)。**遅延ロード**＋可能なら **Web Worker** で実行しUIを止めない。
  WASMは MediaPipe 同様 `public/` 自前ホスト（git管理外、build時コピー）を踏襲。
- `ResultStep` の流れ：正規化 → ランドマーク → `extractLines` →
  - 成功：実測点列＋実測特徴で描画＆診断
  - 失敗：従来のテンプレ追従＋（モック/standard）＋手調整③
- 手調整③は常に補正として残す（自動が外れても救済できる）。

## リスクと正直な限界

- 照明・肌・しわとの区別で**精度に限界**（研究レベルでも難題）。完璧は狙わない。
- **信頼度ゲート＋手調整③** で失敗を救済する設計にする（自動が主、人が補正）。
- 主要線と二次しわの分離が最難関 → ROI＋期待方向＋長さ/濃さで主要線を選ぶ。
- 当面は **基本4線のみ**対象。extra3（太陽/財運/結婚線）は短く薄く誤検出しやすいため後フェーズ。

## 追補：基準ハンド図からの線位置（(u,v) 校正値）

標準的な7大線の解説図（右手・手のひらをカメラへ・親指が左＝負u側）から、
各線の標準位置を本アプリのパーム座標系 `(u,v)` に写し取ったもの。
`lib/lineExtraction.ts` の `CFG`（探索帯・期待位置・弧）はこの表を基準に校正する。

> 座標の約束：u=0 が人差し指‑中指の谷(A)、u=1 が薬指‑小指の谷(B)。
> 親指側は u<0、小指側は u>1。v=0 が指の付け根ライン、v が増えるほど手首へ。

| 線 | 起点 (u,v) | 終点 (u,v) | 検証済み推奨位置 | 形状・膨らみ | CV対象 |
|---|---|---|---|---|---|
| 感情線 | (1.30, 0.10) | (0.10, 0.15) | **v = 0.15〜0.20** | 小指側→人差し指側へほぼ水平。橈側端で指の谷へ上反り。 | ○ 期待 v≈0.175 |
| 知能線 | (−0.10, 0.22) | (0.95, 0.45) | **v = 0.32〜0.38** | 生命線と起点を共有。中央をやや右下がりに横断。 | ○ 期待 v≈0.35 |
| 生命線 | (−0.05, 0.20) | (−0.15, 0.95) | （弧トレース・下記参照） | 親指球を囲む弧。中央で u≈−0.30 まで**親指側へ凸**に膨らむ。 | ○ 弧トレース |
| 運命線 | (0.45, 0.95) | (0.42, 0.15) | **u = 0.48〜0.52** | 手首→中指へほぼ垂直。中央やや橈側。 | ○ 期待 u≈0.50 |
| 太陽線 | (0.75, 0.35) | (0.75, 0.10) | **u = 0.72〜0.78** | 薬指の下に短い縦線。 | 参考のみ(extra3) |
| 財運線 | (0.40, 0.30) | (0.45, 0.10) | **u = 0.88〜0.94** | 小指寄りに短い縦線。 | 参考のみ(extra3) |
| 結婚線 | (1.35, 0.13) | (1.50, 0.13) | **u = 1.28〜1.40** | 小指側エッジに短い横線。 | 参考のみ(extra3) |

- 「検証済み推奨位置」は別アプリで検証した各線の標準位置（横断線=v帯／縦線=u帯）。
  基本4線（感情・知能・運命）は `CFG` の `expected` をこのレンジ中央値に合わせる。
  extra3（太陽・財運・結婚）は参考記録のみ（CV非対象）。
- **CV対象は基本4線のみ**。extra3 は位置を記録するに留め、検出は後フェーズ（誤検出回避）。
- **生命線の弧の向き（決定済み：中央側へ凸）**：基準図では生命線は親指側へ凸（中央で最も負u）だが、
  利用者の実写での見た目を優先し、`lifeArc` は**中央側へ凸（反転, `-0.15 + 0.12·sin`）**を既定とする。
  位置(u,v)の校正は基準図に合わせ済み。基準図どおり親指側へ凸へ戻す場合は `sin` の符号を戻す。
- 感情線の橈側端：基準図は u≈0.10 まで。現状は利用者要望で u≈−0.15 まで延長して運用中。
